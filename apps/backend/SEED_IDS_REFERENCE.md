## Notes

### Consistent Across Runs

The seed uses `upsert` with fixed IDs, so running `npm run prisma:seed` multiple times produces the same IDs.

✅ Safe for tests
✅ Safe for documentation
✅ Safe for API examples

### Per-Seed Randomization

Within the same seed run:

- ✅ Names are randomized (Faker)
- ✅ Phone numbers are randomized
- ✅ Timestamps are randomized (last 30 days)
- ✅ Sentiment scores are randomized
- ✅ Transcripts are randomly selected

This makes the data realistic but varied.

### Reset Changes IDs?

No. Running `npm run prisma:reset` uses the same seed.ts, so IDs remain identical.

## Seed Creation Timestamps

All records are created with `createdAt` timestamps distributed over the **last 30 days**, making it realistic for analytics queries:

```typescript
// Example: Filter calls from last 7 days
const recentCalls = await prisma.call.findMany({
```

## Troubleshooting

### ❌ "ID not found"

Make sure you ran `npm run prisma:seed` first:

```bash
npm run prisma:seed
```

### ❌ "Different IDs each time"

You probably have a modified seed.ts. Check that:
- Company IDs use fixed `where: { id: 'demo-company-*' }`
- User IDs use fixed `clerkId: 'demo-user-*'`

If modified, restore from git:

```bash
git checkout prisma/seed.ts
npm run prisma:reset
```

### ❌ "No data in API response"

Check:
1. Company ID is correct (typo?)
2. User is part of that company
3. Seed ran successfully (`npm run prisma:seed`)
4. Database is not empty (`npm run prisma:studio`)

## Updated Manually?

If you've manually created records in the database, IDs may differ.

**To get real IDs:**

```bash
# View in Prisma Studio
npm run prisma:studio
# or check with psql
psql $DATABASE_URL -c "SELECT id, name FROM companies LIMIT 5;"
```
